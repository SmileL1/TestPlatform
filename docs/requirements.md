# TestPlatform 需求设计文档

> 版本：v2.0 ｜ 更新日期：2026-06-16

本文从需求视角描述系统目标、角色、功能性需求（FR）、非功能性需求（NFR）、典型用例与验收标准。
架构实现见 [架构设计文档](architecture.md)，接口细节见 [详细设计文档](design.md)。

---

## 1. 背景与目标

SmartZaiko 是一套日语仓库管理 WPF 桌面系统，业务流程（如伝票作成/発行/保留）多、表单深、功能键多。
传统手工回归测试耗时且易漏；纯脚本自动化又因界面/数据频繁变动而脆弱、维护成本高。

**TestPlatform 的目标**：让测试人员**无需编码**即可创建、执行、复盘对 SmartZaiko 的 UI 自动化测试，
并借助 LLM 提升用例的鲁棒性与结果判定的可信度。

**衡量目标（北极星）**

- 一名不懂代码的测试人员能在 10 分钟内通过「录制」产出一条可回放用例。
- 同一条用例在界面/数据小幅变动时仍能稳定通过（重试 + AI 验证兜底）。
- 每次执行可逐步回看（工具/参数/结果/AI 思考），失败可定位到具体步骤。

## 2. 角色

| 角色 | 描述 | 主要诉求 |
|------|------|----------|
| **测试人员（主）** | 负责设计并执行回归测试，可能不会写代码 | 录制造例、跑用例、看结果、组计划 |
| **测试负责人** | 安排回归范围、看通过率 | 测试计划、批量执行、历史统计 |
| **平台管理员** | 配置 AI 接口、维护环境 | 设置页配置密钥/模型、清理日志 |

## 3. 功能性需求（FR）

### FR-1 场景管理
- FR-1.1 支持新增/编辑/删除/查询测试场景。
- FR-1.2 场景含：名称、类型（wpf）、目标窗口标题、自然语言描述（AI 目标）、最大步数。
- FR-1.3 场景可定义**参数**（name/label/defaultValue），描述与步骤中以 `{{参数名}}` 引用，运行时注入。
- FR-1.4 场景可定义**结构化验证条件**（见 FR-5）。
- FR-1.5 可一键导入示例场景（基本传票发行/传票保留/新建明细行）。

### FR-2 操作录制
- FR-2.1 指定目标窗口后开始录制，捕获：鼠标点击、表格单元格点击、文本输入、下拉选择、功能键（Enter/Tab/Esc/F1~F12）。
- FR-2.2 录制步骤通过实时通道即时呈现，可在前端**删除/调整/改值**后再保存。
- FR-2.3 可将录制结果**另存为新场景**或**覆盖已有场景**的步骤（保留名称/窗口/断言等配置）。
- FR-2.4 录制需自动过滤无法回放的噪声（只读计算字段、控件模板内部件、类名式名称等）。
- FR-2.5 录制与执行互斥：开始执行前若在录制需自动停止。

### FR-3 执行引擎
- FR-3.1 提供三种模式：`auto`（有录制步骤走结构化，否则 AI）、`structured`（强制回放）、`ai`（强制 LLM 推理）。
- FR-3.2 **AI 推理执行**：LLM 通过工具调用逐步操作界面，工具含点击/输入/选项/读值/表格行列/弹窗检测/断言/完成等。
- FR-3.3 **结构化回放**：按录制步骤顺序重放；单步失败自动重试一次；连续 5 步失败提前终止；
  点击坐标可作为元素定位失败时的兜底。
- FR-3.4 回放中对非弹窗按钮点击后自动检测并关闭应用内弹窗（最多连续 5 层），但录制里显式点了弹窗按钮时尊重录制。
- FR-3.5 可随时取消执行（单次或整个计划）。

### FR-4 AI 截图验证（可选）
- FR-4.1 场景可开启「AI 截图验证」，执行结束对结果界面截图，连同测试目标交多模态模型判定通过/不通过。
- FR-4.2 验证模型独立于操作模型，单独配置（OpenAI 兼容）；未配置则自动跳过、不影响主流程。
- FR-4.3 AI 判定与结构化条件可叠加：启用的判定都通过才算最终通过；AI 还可在 Agent 未显式完成时「翻盘」判通过。

### FR-5 结果判定
- FR-5.1 结构化验证条件支持：`equals/contains/notEmpty`（读控件值）、`exists/notExists`、`textVisible/textNotVisible`、`noDialog/dialogContains/dialogNotContains`。
- FR-5.2 `Expected` 支持 `{{参数}}` 占位。
- FR-5.3 判定优先级：有验证条件/AI 验证 → 按其结果；都没有 → 「所有步骤无失败」。
- FR-5.4 运行结果状态：`running/passed/failed/cancelled`，并给出可读 summary 与失败原因。

### FR-6 测试计划
- FR-6.1 支持创建计划，向计划中批量添加/移除场景并排序。
- FR-6.2 计划可按选定模式批量**顺序**执行，实时展示每个场景状态与整体进度。
- FR-6.3 统计计划通过/失败数，保留计划执行历史（近 20 次）。
- FR-6.4 计划执行可整批取消。

### FR-7 实时监控与历史
- FR-7.1 单次执行实时推送每一步（步号/工具/参数/结果/成功标志/AI 思考/时间戳）与最终结果。
- FR-7.2 录制实时推送每条录制步骤。
- FR-7.3 历史可按场景查询近 100 次执行；可拉取某次执行的完整步骤日志。

### FR-8 设置与运维
- FR-8.1 「设置」页在线管理两套 AI 配置（操作 DeepSeek / 验证 AiVision 的 ApiKey/Model/BaseUrl）、默认目标窗口、日志保留天数。
- FR-8.2 API Key 加密落库，接口只回传「是否已配置」，绝不回传明文；留空表示保持原值。
- FR-8.3 配置优先级：数据库（设置页）> `appsettings.json`。
- FR-8.4 可枚举当前桌面顶层窗口供「目标窗口」选择。
- FR-8.5 可查看 AI 日志用量并手动清理过期日志。

## 4. 非功能性需求（NFR）

| 编号 | 类别 | 要求 |
|------|------|------|
| NFR-1 | 平台 | 后端与被测应用同机运行于 Windows 10/11；目标框架 `net9.0-windows` |
| NFR-2 | 实时性 | 步骤产生到前端可见延迟尽量小（SignalR 实时推送，落库与推送解耦） |
| NFR-3 | 健壮性 | 全局钩子委托防 GC 回收；入库失败不影响实时日志；偶发 UI 时序失败可重试 |
| NFR-4 | 安全 | 敏感密钥加密存储、不回传明文；CORS 限定本地前端来源 |
| NFR-5 | 可观测 | AI 请求/响应留痕（不含图片 base64），按保留天数自动清理 |
| NFR-6 | 易用性 | 不写代码即可造例；录制步骤前端可编辑；前端为编辑风、信息密度适中 |
| NFR-7 | 可维护 | CodeFirst 自动建表 + 启动迁移，降低部署与升级门槛 |
| NFR-8 | 兼容性 | 验证模型走 OpenAI 兼容协议，自动补全 `/v1/chat/completions`，适配多厂商 |

## 5. 典型用例

### UC-1 录制并回放一条用例
1. 测试人员在「操作录制」选择窗口 `SmartZaiko` 开始录制。
2. 在被测应用中完成一次伝票発行操作；步骤实时滚入列表。
3. 删除误操作步骤、必要时把固定值改为 `{{参数}}`，保存为新场景。
4. 回到场景列表点「运行」，选 `structured`，跳转监控页观察逐步回放与最终判定。

### UC-2 用 AI 跑没有录制步骤的场景
1. 新建场景，仅写自然语言目标（如「填车番→加明细→発行并确认成功」）与验证条件。
2. 以 `ai` 模式运行；LLM 逐步 `scan_ui/click/set_text/...`，监控页可见每步思考。
3. 结束后（可选）AI 截图验证复核，给出综合判定。

### UC-3 组计划做批量回归
1. 测试负责人新建计划，加入多个场景并排序。
2. 选模式「运行计划」，监控页看各场景串行执行与通过/失败统计。
3. 失败的场景点开查看其步骤日志定位问题。

### UC-4 配置 AI 接口
1. 管理员在「设置」填 DeepSeek 与（可选）AiVision 的 Key/Model/BaseUrl 并保存。
2. 再次打开仅显示「已配置」标记，不回显密钥；执行时优先生效。

## 6. 验收标准（节选）

- AC-1：导入示例场景后，「基本传票发行」在被测应用就绪时能以 `structured`/`ai` 跑完并给出明确判定。
- AC-2：录制 10 步以内的操作，停止后步骤数与实际操作一致（噪声已过滤），保存后可回放。
- AC-3：开启 AI 截图验证且配置了多模态模型时，失败界面（含错误弹窗）应被判为不通过且给出理由。
- AC-4：在设置页保存 Key 后，`GET /api/settings` 不返回任何密钥明文，仅返回 `configured=true`。
- AC-5：计划批量执行中点「取消」，未开始的场景标记为 cancelled，正在执行的尽快中止。
- AC-6：任一次执行均可在「执行历史」查到，并能拉取完整步骤日志。

## 7. 范围之外（当前版本）

- Web/浏览器自动化（`Type=web` 仅占位，无浏览器驱动）。
- 并行/分布式执行多用例。
- 套件（Suite）维度的完整管理 UI（接口存在但未接通建表与前端）。
- 跨平台运行后端（强依赖 Windows）。
